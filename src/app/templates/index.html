About

GeoBuilder is a tool for writing and solving systems of geometric constraints for olympiad-style geometry problems. The input to GeoBuilder is a problem statement (i.e., system of geometric constraints) written in a custom domain-specific language (DSL). This DSL permits reasoning about three types of geometric objects: points, lines, and circles. GeoBuilder outputs diagrams (i.e., models for the constraint system) that satisfy the problem statement. The user can specify how many such models they would like to generate, and can subsequently download information about these models for inspection. GeoBuilder is also made available as a command-line interface for integration with other tools.

This work was motivated by an ongoing effort to complete the IMO Grand Challenge -- an effort led by Daniel Selsam to build an AI that can win a gold medal at the International Mathematical Olympiad (IMO). The models generated by GeoBuilder are intended to guide solvers for such geometry problems, though it may have uses in other domains (e.g., education, computer-aided design). For technical details, please refer to the arXiv paper.

Tutorial

The following tutorial describes the DSL in which users write GeoBuilder problem statements for model generation. This DSL can broadly be summarized in two parts:

  1. A set of commands to construct a numerical optimization problem to solve for parameterized points, lines, and circles
  2. A set of functions and predicates to define points/lines/circles/numerical values and relationships between them to serve as arguments to the aforementioned commands

All commands, predicates, and function symbols follow Lisp-style s-expression format.

Preliminaries

This tutorial makes common use of the following notation:

  • NAME: A string value that refers to the name of a geometric object (i.e., point, line, or circle). Refers to the name of an object of any type unless specified (e.g., POINT-NAME)
  • VAL: The value of a geometric object or numerical value. A value is denoted either by a NAME or a function (see Functions and Predicates). Like NAME, VAL is used in this tutorial to refer to a value of any type unless specified (e.g., LINE-VAL). Note that names cannot be given to numerical values
  • TYPE: Can take one of three string values: POINT, LINE, or CIRCLE

PRED and FUNC are used as shorthand for PREDICATE and FUNCTION, respectively

.

Commands

GeoBuilder generates models by translating a geometric constraint problem into a numerical optimization problem and solving it with off-the-shelf optimizers. The following four commands reflect this, and comprise the primary component of the DSL

  • COMPUTE: Introduces a geometric object called NAME of type TYPE and assigns it value VAL.

    (COMPUTE NAME TYPE VAL)


    Valid Usage
    • (COMPUTE P POINT (INCENTER A B C))
    • (COMPUTE GAMMA CIRCLE (INCIRCLE A B C))
    • (COMPUTE L LINE (PERPBIS B C))
      • The type of VAL is consistent with TYPE in all above commands
    Invalid Usage
    • (COMPUTE X NUMBER (DIST A B))
      • Numerical values cannot be named
    • (COMPUTE P POINT (INCIRCLE A B C))
      • The type of VAL must match TYPE
    • (COMPUTE P POINT A)
      • VAL cannot be a name (i.e., a string) as duplicate objects are not permitted

    Note that detail on functions used in the above examples (e.g., INCENTER, DIST) is provided in the next section

  • PARAM: Introduces a geometric object called NAME of type TYPE and parameterizes it either by the default parameterization for TYPE or by a supplied parameterization

    (PARAM NAME TYPE OPTIONAL-PARAMETERIZATION)

    Each type has the following default parameterization:
    • POINT: Parameterized by it's x- and y-coordinates
    • LINE: Parameterized by it's coefficients in standard form (ax + by = c)
    • CIRCLE: Parameterized by it's origin and radius (where the origin is parameterized as a point)

    LINE and CIRCLE have no additional parameterizations, while one of the following optional parameterizations can be supplied for a POINT P:
    • (ONLINE P LINE-VAL): Parameterize P on a line
    • (ONSEG P POINT-VAL POINT-VAL): Parameterize P on the segment connecting two points
    • (ONCIRC P CIRC-VAL): Parameterize P on a circle


    Valid Usage
    • (PARAM P POINT)
      • Introduces a point P parameterized by its coordinates
    • (PARAM P POINT (ONLINE P (LINE A B)))
      • Introduces a point P parameterized on the line connecting A and B
    • (PARAM P POINT (ONCIRC (C3 A B C)))
      • Introduces a point P parameterized on the circumcircle of A, B, and C
    • (PARAM GAMMA CIRCLE)
      • Introduces a circle GAMMA parameterized by its origin and radius
    • (PARAM L LINE)
      • Introduces a line L parameterized by it's coefficients in standard form
    Invalid Usage
    • (PARAM X NUMBER)
      • Numerical values cannot be named
    • (PARAM P POINT (INCIRCLE A B C))
      • INCIRCLE A B C is not a valid parameterization of a point
    • (PARAM L LINE (ANYTHING))
      • LINE has no optional parameterizations
    • (PARAM L CIRCLE (ANYTHING))
      • CIRCLE has no optional parameterizations
    • (COMPUTE P POINT (MIDP A B))
      (PARAM P POINT)
      • Cannot introduce the same object twice


    In addition to the standard grammar for parameterizations outlined above, there is an important variation of PARAM to introduce sets of points that form triangles and polygons. These variations take the following form:


    (PARAM (NAME, NAME, ..., NAME) PARAMETERIZATION)

    Note the ommission of type information -- the type of (POINT, POINT, ..., POINT) is implied. While it could be standard to infer TYPE in a COMPUTE or PARAM command when possible, it is generally enforced to improve readability.

    The following values of PARAMETERIZATION are available:
    • TRIANGLE: Parameterize a list of three points as a triangle
    • ACUTETRI: Parameterize a list of three points as an acute triangle
    • (ISOTRI NAME): Parameterize a list of three points as an isosceles triangle. NAME must be in the list of point names being parameterized. For example, (PARAM (A B C) (ISOTRI A)) introduces an isosceles triangle with AB = AC
    • (ACUTEISOTRI NAME): Parameterize a list of three points as an acute isosceles triangle per the above description of an isosceles parameterization
    • (RIGHTTRI NAME): Parameterize a list of three points as a right triangle. NAME must be in the list of point names being parameterized. For example, (PARAM (A B C) (RIGHTTRI A)) introduces a triangle with a right angle at <BAC
    • POLYGON: Parameterize a list of n points as a convex polygon


    Valid Usage
    • (PARAM (A B C) TRIANGLE)
      • Introduces three points, A, B, and C that form a triangle
    • (PARAM (A B C) (ISOTRI B))
      • Introduce three points that form an isosceles triangle with BA = BC
    • (PARAM (A B C D E) POLYGON)
      • Introduces five points, A, B, C, D, and E, that form a convex pentagon
    Invalid Usage
    • (PARAM (A B C D) TRIANGLE)
      • Only lists of points of length three can be parameterized as a triangle
    • (PARAM (A B C) (ISOTRI D))
      • The argument to ISOTRI must be a point in the isosceles triangle (i.e., a point being introduced)
    • (PARAM (A B) POLYGON)
      • A polygon has at least three vertices
    • (PARAM (A B C) (POINT POINT POINT) TRIANGLE)
      • Type information should not be included for commands introducing lists of points that form triangles/polygons
  • ASSERT: Imposes a additional constraint PRED on the system. This command does not introduce any new geometric objects and can only refer to those that have already been introduced by either COMPUTE or PARAM.

    (ASSERT PRED)


    Valid Usage
    • (ASSERT (PARA L1 L2))
      • Assert that line L1 is parallel to line L2
    • (ASSERT (NOT (COLL A B C)))
      • Assert that points A, B, and C are not collinear
    • (ASSERT (EQ 4.0 (DIST A B)))
      • Numerical values find their place in ASSERT. The above asserts that the distance between points A and B is 4.0
    Invalid Usage
    • (ASSERT (PARA L1 L2) (NOT (COLL A B C)))
      • Assertions are made one at a time

    Detail on supported predicates (e.g., PARA, COLL) is provided in the following section

    .
  • CONFIRM: Filter final solutions by those in which PRED is satisfied. Like ASSERT, this command does not introduce any new geometric objects.

    (CONFIRM PRED)

    Usage for CONFIRM is the same as that for ASSERT. The key distinction between ASSERT and CONFIRM is that CONFIRM does not enforce PRED via numerical optimization. This command is most useful for those interested in integrating GeoBuilder with automated theorem provers. As many problem statements are of the form "If P, prove Q," one may wish to optimize directly for P while only confirming that Q holds in the model.
